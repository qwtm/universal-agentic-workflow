Universal Agentic Workflow

Architecture diagrams for the composable, extensible plugin system built on Copilot's agentic primitives — Skills, Agents, MCP Servers, and Hooks — orchestrated through declarative configuration and tool-agnostic archetypes.

Sequence User Request Lifecycle

Traces a user request (e.g. "generate my standup") from intent detection through skill loading, progressive disclosure, MCP data fetching, and structured response delivery.

U-->>R: Free-text activity summary end Note over R,OUT: Phase 4 - Response Generation R->>OUT: Structure into Yesterday / Today / Blockers OUT-->>U: Formatted standup with ticket references
Key insight: The same skill produces identical output structure regardless of mode. MCP connections enrich the data automatically — the skill is tool-agnostic by design, referencing ~~source control not "GitHub."

Extensibility pattern: The 5-phase archetype is itself declarative — each phase's questions are driven by the component-schemas.md reference file. Adding a new primitive type means adding its schema, and the creation workflow automatically incorporates it.

Sequence Plugin Customization Workflow

How cowork-plugin-customizer adapts a tool-agnostic plugin to an organization's specific tools and conventions using ~~placeholder replacement and MCP discovery.

sequenceDiagram autonumber participant U as User participant C as Claude Customizer participant P as Plugin Files participant KMCP as Knowledge MCPs participant REG as MCP Registry participant SC as suggest_connectors U->>C: "Customize engineering plugin for our team" Note over C,P: Phase 0 - Read Plugin and Detect Mode C->>P: Scan for placeholder tokens in skills/ and CONNECTORS.md P-->>C: Found: source-control, project-tracker, chat, monitoring, CI-CD C->>C: Mode = Generic Setup (has placeholders to replace) Note over C,KMCP: Phase 1 - Gather Org Context par Knowledge MCP Queries C->>KMCP: Slack search for jira OR asana OR linear C->>KMCP: Slack search for deploy OR CI OR pipeline C->>KMCP: Notion search for engineering onboarding end KMCP-->>C: Team uses Linear, GitHub, Datadog Note over C: Phase 2 - Build Todo List C->>C: Plan replacements: source-control=GitHub, tracker=Linear, chat=Slack, monitoring=Datadog Note over C,P: Phase 3 - Execute Replacements loop For each placeholder C->>P: Edit SKILL.md: replace category token with real tool name C->>P: Edit CONNECTORS.md: update category mappings end Note over C,SC: Phase 4 - Connect MCPs loop For each identified tool C->>REG: search_mcp_registry(keywords=["linear"]) REG-->>C: Found Linear MCP entry C->>SC: suggest_connectors(directoryUuids=[uuid]) SC-->>U: OAuth prompt for Linear U-->>SC: Authorized end C->>P: Update .mcp.json with connected server URLs Note over C,U: Phase 5 - Package and Deliver C->>P: Package customized plugin C-->>U: Delivered: engineering-yourteam.plugin
Three customization modes: (1) Generic Setup — plugin has ~~ placeholders to replace; (2) Scoped — user targets specific components ("update the standup skill"); (3) General — open-ended modifications. The customizer auto-detects which mode applies.

Sequence Memory Lookup Chain

The two-tier memory system's progressive lookup strategy: hot cache (CLAUDE.md) → deep storage (memory/) → user query. Designed so 90% of lookups resolve from the hot cache.

sequenceDiagram autonumber participant U as User participant C as Claude participant HOT as CLAUDE.md Hot Cache participant GLOSS as memory/glossary.md participant DEEP as memory/projects U->>C: "Ask Todd about the PSR for Oracle" Note over C,HOT: Tier 1 - Hot Cache Lookup C->>HOT: Lookup Todd, PSR, Oracle HOT-->>C: Todd = Todd Chen (Finance VP), PSR = Pipeline Status Report, Oracle = Oracle Systems Q2 deal Note over C: All resolved from hot cache (fast path) C-->>U: Ask Todd Chen about the Pipeline Status Report for the Oracle Systems Q2 deal Note over C,HOT: Scenario 2 - Partial Cache Miss U->>C: "Get Mika's take on DORA for Starfish" C->>HOT: Lookup Mika, DORA, Starfish HOT-->>C: Mika = Mika Patel (Eng Lead) - found. DORA and Starfish - not found. Note over C,GLOSS: Tier 2 - Glossary Lookup C->>GLOSS: Lookup DORA and Starfish GLOSS-->>C: DORA = DevOps Research and Assessment. Starfish - not found. Note over C,DEEP: Tier 3 - Deep Storage C->>DEEP: Search memory/projects/ for Starfish DEEP-->>C: Starfish = microservices migration project, Q3 Note over C,HOT: Promote to Hot Cache C->>HOT: Add Starfish and DORA entries to CLAUDE.md C-->>U: Decoded request with full context
Design goal: CLAUDE.md stays compact (~50-80 lines) covering ~30 people, ~30 terms, and 5-15 active projects. Items promoted/demoted as work context shifts. Tables used for density.
